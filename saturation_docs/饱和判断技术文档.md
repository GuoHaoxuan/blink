## 1. 背景

HXMT（慧眼）卫星高能 X 射线望远镜（HE）在极高计数率事件（如伽马射线暴 GRB）期间，探测器前端电子学的数据缓冲区可能溢出，导致部分物理事例（event）数据丢失。我们将这种状态称为**饱和（saturation）**。

准确识别饱和时间区间对科学分析至关重要：
- 饱和区间内的光变曲线不反映真实源流量
- 需要对饱和区间内的计数率进行死时间校正或标记排除
- 1B 级数据在原始管线中不做饱和标注，需要自行判断

## 2. 硬件架构

### 2.1 数据通路概览

```
NaI/CsI 探测器
    ↓ 模拟信号
前端电子学 (ASIC)
    ↓ 数字事例
FPGA
    ↓ 写入
FIFO A (M67204H, 硬件环形缓冲区)
    ↓ MCU读取
MCU (8051 单片机, PDAUA)
    ↓ 打包写入
FIFO B
    ↓ 1553B总线
星务计算机 → 下传
```

### 2.2 FIFO A — M67204H 芯片

- **类型**：硬件 FIFO，环形缓冲区结构
- **关键特性**：
  - 有 Full 标志位（`FIFOAFull1`, `FIFOAFull2`），由硬件自动置位
  - **满时不覆盖**：当 FIFO 满时，FPGA 端写入被阻塞，新事例直接丢弃
  - MCU 可通过写 `ResetFIFOA = 0x00` 复位整个 FIFO
- **地址映射**：`0xE004`（读）/ `0xE001`（复位）

### 2.3 MCU — 8051 单片机

MCU 是**单线程**处理器，通过主循环轮询处理所有任务。

#### 主循环结构（简化）

```c
while(1) {
    ResetWDI = ~ResetWDI;       // 看门狗喂狗
    SearchStack();               // 处理 1553B 消息
    ReStartTimer();              // GPS 同步
    FIFOAFullReset();            // 检查 FIFO A 是否满，满则复位
    
    if (ReadFlag == 1) {         // 每秒一次
        HandleEnginnerLVDS();    // 工程数据处理 (~2ms)
    }
    
    if (INT1Flag == 1) {         // FPGA 中断触发
        HandlePhysicalLVDS();    // 物理数据处理 (~7ms)
        INT1Flag = 0;
        EX1 = 1;                 // 重新使能中断
    }
    
    Refresh();
}
```

#### 关键时间参数

| 操作 | 耗时 | 说明 |
|------|------|------|
| HandlePhysicalLVDS | ~7ms | 从 FIFO A 读 109 个事例，打包写入 FIFO B |
| HandleEnginnerLVDS | ~2ms | 处理工程遥测数据 |
| FIFOAFullReset | 极短 | 检查 Full 标志，触发复位 |

### 2.4 物理数据处理流程 — HandlePhysicalLVDS

```
1. 复位 FIFO B
2. 写入 CCSDS 包头（6 字节）
3. 循环 109 次：
   a. 搜索 0x5A 起始标记
   b. 从 FIFO A 读取 8 字节事例数据
   c. 直接转写入 FIFO B
4. 写入 UTC 时间尾部（4 字节）
5. 通知 FPGA 开始下一轮 LVDS 传输
```

每个 CCSDS 包 = 6 字节包头 + 872 字节载荷（109 × 8 字节事例）+ 4 字节 UTC 尾 = **882 字节**

## 3. 饱和丢数机制

### 3.1 两种丢数模式

#### 模式一：包内无痕丢数（常见）

```
时间线
─────────────────────────────────────────────
FPGA写入:  ████████████████████████████████████
MCU读取:   ████ HandlePhysicalLVDS (~7ms) ████
FIFO状态:  ░░░░░▓▓▓FULL▓▓▓░░░░░░░░░░░░░░░░░░░
                    ↑ 丢弃    ↑ 恢复
```

**过程**：
1. MCU 正在 `HandlePhysicalLVDS()` 中逐字节读取 FIFO A
2. 高计数率下，FPGA 写入速度 > MCU 读出速度
3. FIFO A 在某一刻被写满，FPGA 无法写入，新事例被丢弃
4. MCU 继续读取，读出一些数据后 FIFO A 不再满，FPGA 恢复写入
5. `HandlePhysicalLVDS()` 完成后回到主循环，`FIFOAFullReset()` 检查时 FIFO A 已经不满 → **不触发复位**

**结果**：
- 丢失少量事例
- CCSDS 包结构完整，包间时间间隔无异常
- **无法通过包间 gap 检测到**

**为什么常见**：
- 只要写速率略高于读速率，FIFO 就会在处理包的 7ms 内短暂达到满状态
- 但因为 MCU 一直在读，很快又恢复了

#### 模式二：整包丢失 / FIFO 复位（较少但影响大）

```
时间线
─────────────────────────────────────────────
FPGA写入:  ██████████████████████████████████████
MCU读取:   ████ Handle ████ FIFOAFullReset() ██
FIFO状态:  ░░░░░▓▓▓▓FULL▓▓▓▓▓▓STILL FULL▓▓▓▓▓▓
                                   ↑ 复位！清空所有数据
```

**过程**：
1. 写入速率**极高**，在 `HandlePhysicalLVDS()` 的 7ms 期间写入量远超读出量
2. MCU 处理完一个包回到主循环时，FIFO A **仍然是满的**
3. `FIFOAFullReset()` 检测到 `FIFOAFull1 && FIFOAFull2 == 0`（Full 标志有效）
4. 执行 `ResetFIFOA = 0x00`，**清空整个 FIFO A**
5. 清除 `INT1Flag`，重新使能外部中断

**结果**：
- 丢失整个 FIFO 缓冲区的数据
- 相邻 CCSDS 包之间出现**大的时间间隔**
- **可以通过包间 gap 检测到**

**为什么影响大**：
- 一次复位丢失大量连续事例
- 在光变曲线上表现为计数率突降或时间间隙

### 3.2 对比总结

| | 模式一：包内无痕丢数 | 模式二：FIFO 复位整包丢失 |
|---|---|---|
| **触发条件** | 写速率 > 读速率，但差距不大 | 写速率 >> 读速率 |
| **丢失量** | 少量事例 | 整个 FIFO 缓冲区 |
| **CCSDS 包影响** | 包结构完整 | 包间出现大 gap |
| **可检测性** | ❌ 无法通过包间 gap 检测 | ✅ 可通过包间 gap 检测 |
| **频率** | 饱和期间常见 | 极高计数率时出现 |
| **影响程度** | 较小 | 显著 |

## 4. 饱和区间检测算法

### 4.1 基本思路

通过检测 CCSDS 包之间的**时间间隔异常**来识别发生了 FIFO 复位（模式二）的时间区间。

### 4.2 算法流程

```
┌─────────────────────────────────┐
│  1. 解析所有 CCSDS 包            │
│     parse_events() → 事例列表    │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  2. 时间重建                     │
│     reconstruct_with_wrap_tracking() │
│     → 每个包的 MET 时间列表       │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  3. 计算每个包的时间范围          │
│     PackInfo { min_time, max_time } │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  4. 按 min_time 排序             │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  5. 滑窗检测包间 gap             │
│     gap = next.min_time          │
│         - current.max_time       │
│     if gap > threshold → 饱和区间 │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  6. 多机箱合并（A/B/C 取并集）   │
│     任一机箱饱和 → 该时段判定饱和 │
└─────────────────────────────────┘
```

### 4.3 Gap 阈值讨论

当前实现使用固定阈值 `GAP_THRESHOLD = 6.9ms`，但这**不够准确**。

正常包间隔取决于实时事例率 R：

$$T_{packet} \approx \frac{109}{R}$$

| 事例率 R (evt/s) | 正常包间隔 | 6.9ms 阈值表现 |
|------------------|-----------|---------------|
| 10,000 | ~10.9ms | ⚠️ 误判：正常间隔已超过阈值 |
| 15,000 | ~7.3ms | ⚠️ 边界情况 |
| 20,000 | ~5.5ms | ✅ 能区分 |
| 50,000 | ~2.2ms | ✅ 灵敏度足够 |

**改进方向**：
- 使用自适应阈值，基于局部事例率动态计算
- 或直接对比相邻包间隔与该时段平均包间隔的偏离程度


## 4. 时间重建算法

### 1. 硬件背景与关键常量

### 数据通路

```
NaI/CsI 探测器 → ASIC → FPGA → FIFO A (M67204H) → MCU (8051) → FIFO B → 1553B → 下传
```

MCU 主循环 (`PDAUA.c`) 逐事件从 FIFO A 读出，填装 CCSDS 包 (882 字节 = 6 头 + 872 载荷(109×8) + 4 UTC 尾)。

### 常量

| 名称 | 值 | 含义 |
|------|-----|------|
| `PTIME_MOD` | 524288 (2^19) | ptime 计数器模值 |
| `WRAP_PERIOD` | 1.048576s | ptime 回绕周期 = PTIME_MOD × 2μs |
| `MET_CORRECTION` | 4.0s | 1B→1K 经验时间校正 |
| `WRAP_THRESHOLD` | 10000 ticks (20ms) | 正常路径 wrap 判定阈值 |
| `ANCHOR_RECENT_PKT_LIMIT` | 35 包 | 锚点"新鲜"判定阈值 |
| `MIN_EVENTS_FOR_MEDIAN` | 50 | 包级 median 可信所需最少有效事件 |
| ptime 分辨率 | 2μs/tick | |
| CRC | 4-bit | 1/16 概率碰撞通过 |

### SEC 锚点

硬件每秒插入一个 `Pack::Second` 事件，记录 `(stime, ptime)`。`stime + offset = MET`。SEC 事件的 `(MET, ptime)` 构成时间重建的锚点 (anchor)。

### 两种丢数模式

1. **包内无痕丢数（常见）**：写速率略>读速率，FIFO 短暂满后恢复，MCU 回到主循环时已不满→不复位。无法检测。
2. **FIFO 复位整包丢失**：写速率>>读速率，HandlePhysicalLVDS 结束后 FIFO 仍满→FIFOAFullReset() 触发复位清空。包间出现大 gap，可检测。

---

### 2. 核心问题：ptime 回绕次数 n_wraps 的确定

每个事件只有 19-bit 的 `ptime`，需要确定它相对于锚点经历了多少次完整回绕。

**通用公式**：
```
raw_delta = event.ptime - anchor_ptime
MET = anchor + (raw_delta + n_wraps × PTIME_MOD) × 2μs + MET_CORRECTION
```

常规条件下 n_wraps 容易确定。但在 GRB 饱和期间：
1. **FIFO 拥塞**：事件滞留导致 `utc_tail` 严重滞后于事件真实时间
2. **FIFO 复位**：复位前最后接受的 SEC 仍是 MCU 当前锚点，直到下一个整秒边界 FPGA 注入新 SEC，这中间的事件都用过期锚点，路径 2 的 n_base 取整可能差 ±1
3. **ptime 回绕边界**：过期锚路径估算 n_base 时用 round() 取整，当包的 median ptime 经过 anchor_ptime 附近时取整翻转 ±1

---

### 3. Pass 1：逐包时间重建

函数 `reconstruct_with_wrap_tracking()` 对每个 CCSDS 包逐事件重建。三条路径按优先级：

### 路径 1：Wrap Tracking（FIFO 拥塞专用）

**触发条件**：`wrap_tracking_active && !anchor_is_recent && elapsed > WRAP_TRACKING_ELAPSED_THRESHOLD`

当锚点变"陈旧"（距 SEC 超过 35 包）且不在 FIFO 复位后，用 median ptime 跟踪回绕次数。

**核心原理**：FIFO 拥塞期间，MCU 顺序读 FIFO，连续包的 median ptime 单调递增。当 median 回绕（大负 delta），`congestion_wrap_count += 1`。这给出正确的 n_wraps 而不依赖 utc_tail。

**初始化**：SEC 被接受时 `congestion_wrap_count=0`，`prev_median_ptime=当前包median`。在 35 包 "recent" 期间不更新 `prev_median_ptime`，确保激活时首次比较跨越所有 recent 包。

**Phase Correction（相位修正）**：

`congestion_wrap_count` 跟踪的是 median ptime 的回绕次数，但时间计算基于 `anchor_ptime`（SEC 的 ptime）。当两者差距超过半周期时，median 看到的回绕比实际需要的多/少 1 次。

```
delta = anchor_median - anchor_ptime
if delta > PTIME_MOD/2:  phase_correction = -1  (median 多算 1 次)
if delta < -PTIME_MOD/2: phase_correction = +1  (median 少算 1 次)
corrected_wrap_count = congestion_wrap_count + phase_correction
```

**时间计算**：使用 `compute_met_with_base_wraps(ptime, anchor_ptime, anchor, corrected_wrap_count, median)`

```
delta_from_median = ptime - median_ptime
wrap_adjust = if delta < -PTIME_MOD/2 { +1 } else if delta > PTIME_MOD/2 { -1 } else { 0 }
total = raw_delta + (corrected_wrap_count + wrap_adjust) × PTIME_MOD
MET = anchor + total × 2μs + MET_CORRECTION
```

这消除了旧 stale path 的逐事件模糊性——当 FIFO 延迟 ≈ (k+0.5)×WRAP_PERIOD 时，旧方法对同包内事件可能分配不同 n_wraps。

### 路径 2：包级过期锚路径

**触发条件**：`!anchor_is_recent && !use_wrap_tracking && elapsed ≥ 1.5×WRAP_PERIOD`

FIFO 复位后锚点过期多个 wrap 周期，但事件是新鲜的（无 FIFO 延迟）。用 `estimate_packet_wraps` 对包的 median ptime 做 best-of-3 估算 `n_base`，再用 `compute_met_with_base_wraps` 处理包内事件。

### 路径 3：正常路径（阈值法）

**触发条件**：`anchor_is_recent` 或 `elapsed < 1.5×WRAP_PERIOD`

锚点新鲜，至多 1 次 wrap。用阈值法判定：

```rust
let raw_delta = ptime as i64 - anchor_ptime as i64;
let adjusted = if raw_delta < -WRAP_THRESHOLD {
    raw_delta + PTIME_MOD  // 正向 wrap
} else if raw_delta > (PTIME_MOD - WRAP_THRESHOLD) {
    raw_delta - PTIME_MOD  // 锚点已跨过回绕边界而事件尚未，同为正向 wrap
} else {
    raw_delta
};
MET = anchor + adjusted × 2μs + MET_CORRECTION
```

**优势**：不依赖 utc_tail，不受滞后影响。
**局限**：当 `|raw_delta| < WRAP_THRESHOLD` 且事件实际跨 wrap 时会误判（Pass 3 修正）。

### 4. Pass 1 补充：FIFO 复位后后向 SEC 修正

### 问题

FIFO 复位清空了 FIFO 中的所有数据，复位后没有旧 SEC 混入。真正的问题是：复位前最后接受的 SEC 仍然是 MCU 的当前锚点。直到下一个整秒边界 FPGA 注入新 SEC，这中间的事件都用该**过期锚点**。旧锚点的 `(MET, ptime)` 本身是正确的，但距现在已过数秒（多个 wrap 周期），路径 2 用 `utc_tail + round()` 估算 `n_base` 时取整可能差 ±1，导致事件时间系统性偏移约 1 个 WRAP_PERIOD。

**表现**：FIFO 复位后 ~100 包的事件被放到比正确位置早 ~1.05s 的位置。例如 Box B T+249 区域 ~9500 事件缺失（被错放到 T+248）。

### 核心思路：前向过期锚 + 后向新鲜锚

FIFO 复位后，过期锚点持续生效直到新鲜 SEC 到达。用新鲜 SEC 作为锚点反向修正之前的事件。

### 算法

#### 状态管理

- `anchor_post_fifo_reset`: FIFO 复位后首个 SEC 被接受时置 true
- `fifo_reset_pending`: 缓冲复位后的事件 `(pkt_idx, evt_idx, ptime, utc_tail)`
- 新鲜 SEC 判定：`!anchor_is_recent`（过期锚点在前 1-2 包内被接受，仍属 "recent"）

#### 事件分类与修正

当新鲜 SEC 到达时，对缓冲的事件执行三阶段修正：

**Phase 1 — 置信度分类**：用 `estimate_packet_wraps_signed` 计算每个事件的后向 wrap 数和置信度 margin。
- `margin > 0.05`：**可确定事件**，直接用后向 SEC 计算 MET
- `margin ≤ 0.05`：**可疑事件**，进入 Phase 2

**Phase 2 — 前后对照**：对可疑事件，比较两个方向的估算：
- `biased_met`：过期锚点已计算的 MET（在 result 中）
- `met_base`, `met_alt`：后向 SEC 的两个候选（相差 ±WRAP_PERIOD）
- 选择与 `biased_met` 最接近的后向候选

**原理**：过期锚点和新鲜锚点有不同的 anchor_ptime，因此它们的 wrap 边界位置不同。当一个方向模糊时，另一个方向通常能给出有偏但方向正确的估计。

**Phase 3 — 应用修正**：用 `compute_met_with_base_wraps` 将修正后的 wrap 数应用到包内所有事件。

#### `estimate_packet_wraps_signed`

允许负 wrap 数（事件在锚点之前），返回 `(n_wraps, margin)`：

```
n_est = round((elapsed - raw_delta × 2μs) / WRAP_PERIOD)  // 无 max(0) 限制
margin = (second_best_err - best_err) / (2 × WRAP_PERIOD)  // 0=模糊, 0.5=确定
```

#### `compute_met_backward`

从未来锚点反向计算，支持多 wrap 跨度：

```
raw_met = future_anchor_met + (ptime - future_ptime) × 2μs + MET_CORRECTION
k = round((raw_met - approx_met) / WRAP_PERIOD)
result = raw_met - k × WRAP_PERIOD
```

### 已知局限：双重模糊

当过期锚点和新鲜锚点的 wrap 边界恰好重合时（两者的 anchor_ptime 差异使得同一批事件在两个方向上都处于 wrap 边界），前后对照无法消歧。表现为 `err_a ≈ err_b ≈ WRAP_PERIOD`。

例：Box B T+277/278 约 3000 事件仍有偏移，待后续改进。

### 5. Wrap Tracking 的安全机制

### FIFO 复位检测 (`fifo_reset_no_wt`)

当 `utc_tail` 向前跳跃 > 3s 时判定为 FIFO 复位（MCU 清空 FIFO）。复位后：
- 事件是新鲜的（无 FIFO 延迟），stale path 的 utc_tail 估算正确
- 拥塞期的 wrap tracking 无效，重置所有状态
- 设 `fifo_reset_no_wt=true` 阻止 wrap tracking 重新激活
- 直到新 SEC 锚点被接受时 `fifo_reset_no_wt` 重置

### 损坏包过滤 (`MIN_EVENTS_FOR_MEDIAN`)

CCSDS 包 109 个事件中只有少数通过 4-bit CRC 时，说明包数据损坏。这些事件的 ptime 不可靠（随机数据碰巧通过 CRC），其 median 可能严重偏离真实值。

保护措施：
- 有效事件 < 50 的包跳过 wrap 检测（不触发 WRAP_INC）
- 不更新 `prev_median_ptime`（避免污染后续比较基准）

### 仅递增规则（无 WRAP_DEC）

ptime 由硬件计数器产生，在 MCU 顺序读取 FIFO 时严格单调递增，不存在物理上的反向回绕。阈值法的两种判定（`raw_delta < -θ` 和 `raw_delta > P_MOD - θ`）都是正向回绕在不同 anchor 位置下的表现。因此 wrap tracking 只检测 WRAP_INC（median 大负 delta），不检测 WRAP_DEC。

---

### 6. Pass 2：FIFO 复位后时间倒退修正 (`fix_wrap_reversals`)

### 问题

MCU 写入顺序本来就是时间顺序（FIFO 严格先进先出）。Pass 2 修正的不是文件乱序，而是过期锚路径的 `n_base` 取整偏差，使一批包被放到高一个 WRAP_PERIOD 的位置。

**表现**：一批包的时间比后续包高 ~WRAP_PERIOD（时间倒退）。

### 算法

1. 构建"干净包"序列（span < 0.3s）
2. 扫描相邻干净包间的**反向跳跃** ≈ -WRAP_PERIOD
3. 从跳跃点反向查找属于高层级的包批次
4. **Gap 判据**区分真假倒退：
   - 批次前有大 gap (> 0.3s) → FIFO 复位后的真错位 → 整批下移 -WRAP_PERIOD
   - 批次前无 gap（平滑衔接） → 纯文件重排的假倒退 → 跳过

---

### 7. Pass 3：ptime 回绕边界修正 (`fix_wrap_boundary_dips`)

### 问题

Pass 3 修正的凹坑不是来自阈值法的死区。ptime 严格递增意味着对新鲜锚点，`raw_delta < 0` 一定是 wrap（无歧义）。凹坑的真正原因是：过期锚路径估算 `n_base` 时用 `round()` 取整，当包的 median ptime 经过 `anchor_ptime` 附近时，`(p_median - p_anchor)` 跳变导致分子变化 ~1.05s，取整翻转 ±1。

**表现**：在 ~WRAP_PERIOD 间隔处出现 "dip"——一小批包的时间比前后邻居低一个 WRAP_PERIOD。

### 算法

#### Step 1：Dip Batch 检测与修正

扫描正向跳跃 ≈ +WRAP_PERIOD，反向查找被 HIGH 包"夹心"的 LOW 批次，整批上移 +WRAP_PERIOD。

#### Step 2：混合包修正

对 span ≈ WRAP_PERIOD 的包，按邻居层级判定多数/少数簇，将少数簇对齐（±WRAP_PERIOD）。

### Pass 2 与 Pass 3 的互补关系

Pass 2 修偏高批次（下移 -WRAP_PERIOD），Pass 3 修偏低批次（上移 +WRAP_PERIOD），是对称互补的操作。Pass 2 需要 gap 判据（区分真错位和文件重排产生的假倒退），Pass 3 不需要（凹坑不会是文件重排）。

---

### 8. Pass 4：分段排序

Pass 4 排的不是文件中的原始包乱序（MCU 写入顺序就是时间顺序），而是后向 SEC 修正改变 MET 后在修正区域和相邻 wrap tracking 区域交界处产生的局部时间重叠。对非后向修正区域的连续段分别排序，后向修正区域保持不动。

---

### 9. 处理管道总结

```
CCSDS 包序列
    │
    ▼
Pass 1: 逐包时间重建 (三路径 + FIFO复位后向修正)
    │   路径1 Wrap Tracking: median 追踪 + phase correction
    │   路径2 Stale Path: utc_tail 辅助 best-of-3
    │   路径3 Normal Path: 阈值法, ≤1 wrap
    │   补充: FIFO复位后向SEC修正 (三阶段: 分类→前后对照→应用)
    │
    ▼
Pass 2: fix_wrap_reversals (逐包批次)
    │   检测反向跳跃 ≈ -WRAP_PERIOD
    │   Gap 判据区分真假倒退
    │
    ▼
Pass 3: fix_wrap_boundary_dips (逐包批次)
    │   Step 1: 检测正向跳跃, 修正 dip batch
    │   Step 2: 修正混合包内少数簇
    │
    ▼
Pass 4: Segmented sort (按需)
    │   后向修正产生的局部时间重叠 → 分段排序
    │
    ▼
输出: Vec<Vec<f64>> — 每包的重建 MET 时间列表
```

### Box C T+328 空洞 — WRAP_DEC 移除

### Box B T+261-262 空洞 — 损坏包 median
**根因**：pkt 39981 损坏，109 事件仅 4 个通过 CRC。随机 ptime 的 median=66592（真实≈355000），误触发 WRAP_INC，wraps 从 2 跳到 3。
**修复**：MIN_EVENTS_FOR_MEDIAN=50，有效事件不足的包跳过 wrap 检测。

### Box A T+550 空洞 — Phase correction
**根因**：SEC 的 anchor_ptime=49418 近 0，但该包 median=521070 近 PTIME_MOD。Median 回绕 1 次（521070→236896），但 anchor_ptime 到 236896 不需要回绕。Wrap tracking 多算 1 次。
**修复**：记录 anchor_median_ptime，计算 phase_correction(-1/0/+1)，corrected_wrap_count = congestion_wrap_count + phase_correction。

### FIFO 复位后 wrap tracking 误激活 — fifo_reset_no_wt
**根因**：FIFO 复位后事件是新鲜的，wrap tracking 不应重新激活。
**修复**：UTC_JUMP 时设 fifo_reset_no_wt=true，直到新 SEC 锚点接受时重置。

### FIFO 复位后过期锚点 — 后向 SEC 修正
**根因**：FIFO 复位清空了所有数据，但复位前最后接受的 SEC 仍是 MCU 当前锚点（过期锚点）。旧锚点距现在已过数秒，路径 2 的 n_base 取整差 ±1，导致事件偏移 ~1 WRAP_PERIOD。
**修复**：检测 FIFO 复位后的过期锚点，缓冲后续事件。等新鲜 SEC 到达后用三阶段后向修正（置信度分类 → 前后对照 → 应用）。

### 12. 已知局限

1. **4-bit CRC 碰撞**：高计数率时 ~6% 随机字节序列可能通过 CRC 校验，产生幽灵事件。
2. **无痕包内丢数**：FIFO 短暂满后恢复，丢失事件无法检测或恢复。
3. **wrap 边界残余**：WRAP_PERIOD ≠ 1.0s，极少数 raw_delta=0 的事件（概率 ~1/500000）无法判定归属。
4. **饱和边界差异**：1B 和 1K 在 FIFO 复位转换处的事件来源不同，存在固有差异。
5. **双重模糊**：当过期锚点和新鲜锚点的 wrap 边界重合时，前后对照无法消歧（Box B T+277/278）。

## 8. 静默丢数检测（包内无痕丢数）

### 8.1 原理

模式一（包内无痕丢数）无法通过包间 gap 检测，但可以通过**包内事件间隔统计异常**来发现。

在 FIFO 拥塞期间，MCU 读速率（~15797 evt/s）成为瓶颈，事件以近乎均匀的间隔被读出。如果 FIFO 在读取过程中短暂满溢后恢复，丢弃的事件会在包内产生一个异常大的时间间隔。

### 8.2 检测方法

对每个高事件率包（包内事件率 > MCU 读速率 15797 evt/s）：

1. 计算包内事件间隔序列 `dt[i] = t[i+1] - t[i]`
2. 用过滤后的间隔（< 1ms）估算包内真实事件率 `λ = 1 / mean(dt_filtered)`
3. 对每个间隔计算泊松概率 `log₁₀(p) = -λ × dt / ln(10)`
4. 若 `log₁₀(p) < -10`，标记为静默丢数可疑位置

**关键改进**：lambda 必须用过滤后的间隔估算，否则大 gap 本身会拉低 lambda，导致同包内其他较小的异常间隔漏检。

### 8.3 检测结果

| GRB | Box A | Box B | Box C | 合计 suspects | 独立包数 |
|-----|-------|-------|-------|--------------|---------|
| 200415A | 1 | 0 | 0 | **1** | 1 |
| 221009A | 42 | 42 | 39 | **138** | ~90 |
| 260226A | 0 | 0 | 0 | **0** | 0 |

### 8.4 典型特征

- **位置集中在包头/包尾**：大多数异常间隔出现在 `evt_gap_idx=0`（包头）或 `evt_gap_idx=106`（包尾），对应 MCU 在包边界处开始/结束读取 FIFO 的时刻，此时 FIFO 写入积压最容易发生满溢
- **间隔规模**：典型 500μs ~ 6000μs（正常间隔 ~10-60μs），对应丢失数十到数百个事件
- **与 FIFO 复位的关系**：静默丢数集中在 FIFO 复位（模式二）密集区附近，说明两种模式是同一拥塞过程的不同严重程度表现
- **GRB 强度相关**：221009A（史上最亮 GRB）有大量静默丢数，260226A（较弱）无静默丢数

## 9. 饱和区域光变曲线重建

### 9.1 目标

在 FIFO 复位导致的数据丢失区间（gap）内，估算丢失的事件数量并重建光变曲线的时间轮廓。

### 9.2 FIFO 复位丢数模型

#### 9.2.1 单次 FIFO 复位的数据丢失

每个 FIFO reset gap（相邻两包之间的时间间隙）对应**恰好一次** FIFO 复位。理由：FIFO 复位后清空，重新填满需要 455/R_true 秒（即使 R_true=50000 也需 ~9ms），而主循环一轮只要微秒级，不可能在一轮内再次填满触发复位。

丢失事件数：

$$N_{lost} = R_{true} \times T_{gap}$$

其中：
- $R_{true}$：真实事件率
- $T_{gap}$：gap 持续时间（下一包首事件 - 上一包末事件）

注意：FIFO 中被清空的 ~455 个事件的时间戳落在 gap 时间范围内，已被 $R_{true} \times T_{gap}$ 包含，不应重复计算。

#### 9.2.2 真实率 R_true 的估算

**关键物理约束**：在 FIFO 持续满载时，MCU 每读走一个事件空出一个位置，FPGA 立刻写入当前时刻的新事件。因此 FIFO 中相邻事件的 ptime 间距 ≈ 1/R_read（MCU 读速率），而非 1/R_true。包跨时 T_span ≈ 109/R_read，**不能直接用于估算 R_true**。

但 FIFO 复位后，FIFO 清空，新事件直接进入，不存在满载阻塞。因此 **post-reset 包**（gap 后第一个包）的事件 ptime 间距反映 R_true：

$$R_{true} \approx \frac{109}{T_{span,post-reset}}$$

实测验证（221009A Box A）：pre-reset 包与 post-reset 包的 span 比值均值 ~0.93，说明 R_true 仅比 R_read 高约 10-20%。

#### 9.2.3 三机箱交叉参考

三个 box（A/B/C）有独立的 FIFO 和 MCU，但观测同一 X 射线源，计数率成比例。当一个 box 饱和时，其他 box 可能仍有完好数据。

221009A 统计：

| 情况 | Box A | Box B | Box C |
|------|-------|-------|-------|
| 另外两个 box 都未饱和 | 49.2% | 38.3% | 46.8% |
| 至少一个 box 未饱和 | 92.0% | 92.0% | 92.6% |
| 三个同时饱和 | 8.0% | 8.0% | 7.4% |

92% 的 gap 至少有一个参考 box 可用。

### 9.3 三种饱和区域及重建策略

根据饱和程度不同，分为三种情况，各有对应的检测和重建策略：

#### 9.3.1 轻度饱和 — 静默丢数（模式一）

**判定条件**：邻居包事件率 > MCU 读速率（15000 evt/s）

**物理机制**：FIFO 短暂满溢 → 集中丢几个到几十个事件 → 恢复。丢失是突发性的，表现为包内异常大间隔。

**检测方法**：
1. 对每个高率包，用过滤后间隔（< 1ms）估算 $\lambda$
2. 对拥塞宽包（span > 邻居中位 span × 3），用邻居事件率做 $\lambda$
3. 泊松检验：$\log_{10}(p) = -\lambda \cdot \Delta t / \ln(10) < -10$ → 标记为静默丢数

**重建方法**：
- 总量：$N_{lost} = \lambda \cdot \Delta t - 1$
- 形状：交叉参考其他未饱和 box 的事件分布，用 shape bin (1ms) + 均值填充空 bin
- 参考覆盖率 < 30% 时直接均匀分配

#### 9.3.2 深度饱和 — 包级修正

**判定条件**：邻居包事件率 < MCU 读速率（15000 evt/s），即所有邻居都是拥塞包

**物理机制**：MCU 每 ~8ms 读一次 FIFO，读取 ~109 个事件在 ~0.5ms 内完成（burst），其余 ~7.5ms 为主循环开销期间。FIFO 在此期间持续满载，新事件无法写入被丢弃。

**检测方法**：通过邻居率判定。深度饱和区不做逐间隔的泊松检测（因为 $\lambda$ 无法从数据可靠估算）。

**重建方法**：
- 用包内 burst 部分（间隔 < 1ms）的计数率作为 $R_{true}$
- $N_{fill} = R_{true} \times T_{gap,intra-packet}$（gap = 包内 > 1ms 的大间隔总时长）
- 补全事件均匀分配在包内各个 gap 间隔中

#### 9.3.3 FIFO 复位 — 整包丢失（模式二）

**检测方法**：包间 gap > baseline × 100（GAP_FACTOR），且最大事件率 > 15000。对深度饱和区，用附近包的率代替相邻包的率判断。

**R_true 估算**：
- 非深度饱和：$R_{true} = 109 / T_{span,post-reset}$（post-reset 包的跨时）
- 深度饱和：用邻近包的 burst rate（包内过滤间隔的率）

**重建方法**：
- 总量：$N_{lost} = R_{true} \times T_{gap}$
- 形状函数：汇总所有参考 box 的事件构建 shape bin
  - 跳过参考 box 在不可信区间内的 bin（FIFO reset gap + 拥塞宽包 + 包内异常间隔）
  - 参考覆盖率 < 30% → 均匀分配
  - 参考覆盖率 ≥ 30% → 用有值 bin 的均值填充空 bin
- 归一化到 $N_{lost}$ 后按形状分配事件

### 9.4 不可信区间检测

交叉参考的质量取决于参考 box 的数据是否可靠。以下时间区间被标记为不可信，参考时跳过：

1. **FIFO reset gap**：整包丢失的时间段，无事件
2. **拥塞宽包**：包跨时 > 邻居中位跨时 × 3，事件分布反映 MCU 读取模式而非真实光变
3. **含异常间隔的包**：包内有泊松异常间隔（用邻居率做 $\lambda$ 判定），整个包的事件分布不可信

### 9.5 处理原则

1. **三种重建独立进行**，都只用原始观测事件做交叉参考，不互相影响
2. **总量由物理模型决定**（$R_{true} \times T_{gap}$ 或 $\lambda \times \Delta t - 1$），形状由交叉参考或均匀分配决定
3. **归一化保证总量守恒**：无论形状函数如何，补全事件总数不变

### 9.6 实现

Rust CLI 命令：

```
HXMT_1B_DIR=data/1B ./target/release/blink_cli <datetime> --reconstruct <center_met> <half_window> [bin_width]
```

输出 CSV 格式：`box,bin_center,observed,reconstructed,filled_gap,filled_sd`

- `filled_gap`：FIFO reset gap 补全的事件率
- `filled_sd`：静默丢数 + 深度饱和修正的事件率

核心函数：
- `detect_silent_drops()`：泊松方法检测包内异常间隔（轻度饱和区）
- `reconstruct_silent_drops()`：交叉参考填充静默丢数
- `reconstruct_deep_saturation()`：burst rate 填充深度饱和包内 gap
- `reconstruct_gaps()`：交叉参考填充 FIFO reset gap
- `detect_unreliable_intervals()`：标记不可信时间区间

### 9.7 局限性

1. **包间中等 gap 漏检**：GAP_FACTOR=100 阈值下，10x~50x 的包间 gap 不会被检测为 FIFO reset，在图上表现为窄缝
2. **深度饱和 R_true 精度**：burst rate 反映 MCU 读速率与 FIFO 写入速率的耦合，可能不等于真实源率
3. **三 box 同时饱和**：~8% 的 gap 无参考 box，走均匀分配
4. **FIFO 容量**：455 事件（M67204H 4096 字节）用于一致性检查，不直接影响重建总量

## 附录 A：CRC 碰撞幽灵事件

## 1. 背景：HE 载荷的 4-bit CRC

慧眼 HE 载荷每个 CCSDS 科学数据包包含 109 个事例（event slot），每个事例 8 字节。
事例的最后 4 bit 是 CRC 校验码，用于检测数据传输错误。

```
事例格式 (8 字节):
  byte[0]     : channel (能道)
  byte[1-3]   : 依类型不同含义不同
  byte[4-6]   : ptime 的高位编码
  byte[7]     : [7:6]=ptime低2位, [5:4]=类型标识, [3:0]=CRC校验
```

CRC 校验流程：
```
computed_crc = crc_check(row[0..7])   // 4-bit 结果
stored_crc   = row[7] & 0x0F
if computed_crc == stored_crc → 事例通过 CRC，被认为是有效数据
else → 标记为 Pack::Error，丢弃
```

### 4-bit CRC 的固有缺陷

4-bit CRC 只有 16 种可能值。当一个事例的数据被损坏（比如 FIFO 读写冲突、
电磁干扰等导致的 bit 翻转），损坏后的数据经过 CRC 计算，有 **1/16 = 6.25%**
的概率恰好等于存储的 CRC 值。

这种情况下，一个**完全错误的事例会被误判为有效数据**——我们称之为
**"CRC 碰撞幽灵事件"（CRC collision ghost event）**。

## 2. 正常情况下为何不是问题

在正常计数率（~几百 evt/s）下：
- 每个包 109 个事例中，CRC 错误通常 < 5 个
- 碰撞通过的幽灵 ≈ 5 × (1/16) ≈ 0.3 个/包
- 偶尔一个幽灵事件不影响整体重建

## 3. 高计数率下的幽灵泛滥

在极端高亮爆发（如 GRB 260226A，T+19~21s）下：
- 事件产生速率远超 FIFO 读取速率
- FIFO 读写冲突频繁，大量事例数据损坏
- **109 个事例中 100+ 个 CRC 错误**（错误率 >90%）

此时幽灵数量：
```
CRC 错误事例 ≈ 100
碰撞通过数   ≈ 100 / 16 ≈ 6 个
真实通过数   ≈ 109 - 100 = 9 个
```

在 9 个"通过 CRC"的事例中，约 6 个是幽灵！**多数"有效"事例实际是垃圾数据。**

## 4. 幽灵事件如何破坏时间重建

### 4.1 幽灵事件的特征

幽灵事件的 8 字节数据完全是损坏后的随机值，因此：
- **channel**：随机（0~255）
- **ptime**：随机（0~524287）
- **类型标识**：随机（可能被误判为 EVT 或 SEC）

### 4.2 对顺序追踪算法的毒害

WrapTracker 的核心逻辑是通过检测 ptime 回落来判定 wrap：

```
若 ptime < prev_ptime 且差值 > HALF_MOD (262144)
→ 判定发生了一次 wrap，wrap_count += 1
```

当一个幽灵事件携带随机 ptime 被追踪时：

**场景**：prev_ptime = 400000（真实值，接近 PTIME_MOD 上限）
- 幽灵 ptime = 50000（随机小值）
- 差值 = 400000 - 50000 = 350000 > HALF_MOD
- **误判为 wrap！** wrap_count 错误地 +1

此后所有真实事件的 MET 都会偏移 +1.048576s。

更严重的是，幽灵的随机 ptime 成为新的 prev_ptime，可能触发连锁错误：
后续真实事件的 ptime 与这个垃圾 prev_ptime 比较，可能再次误判 wrap。

### 4.3 实际观测到的现象

GRB 260226A 中：
- **Box B (T+19~21s)**：~8000 个 1B 事件被整体偏移 ~1.048s
- **Box C (T+23~26s)**：类似偏移
- 1B 光变曲线相对 1K 出现明显的时间位移
- scatter 图中可见事件"条带"错位

## 5. 为什么旧的 floor 公式对此免疫

旧的 `compute_met` 函数使用 per-event floor 公式：

```rust
let n_wraps = ((utc_tail - anchor - WRAP_PERIOD - raw_delta_seconds) / WRAP_PERIOD)
    .floor()
    .max(0.0) as i64;
```

每个事件独立用 utc_tail 估算 wrap 数，**不依赖 prev_ptime**。
幽灵事件的垃圾 ptime 只影响自身的 MET 计算，不会传播到后续事件。

但 floor 公式在 wrap 边界处不稳定（同一包内相邻事件可能得到不同的 n_wraps），
导致 GRB 221009A 出现包时间重叠问题。

## 6. 解决方案：混合追踪 + floor 校准

### 6.1 核心思路

将两种方法的优势结合：
- **正常区域**：使用顺序追踪（WrapTracker），精确处理 wrap 边界
- **高错误率区域**：跳过追踪，避免幽灵污染
- **区域过渡**：用 floor 公式重新校准 wrap_count，然后恢复顺序追踪

### 6.2 实现细节

#### WrapTracker 增加 `needs_recalibration` 状态

```rust
struct WrapTracker {
    anchor: f64,
    anchor_ptime: u64,
    prev_ptime: u64,
    wrap_count: i64,
    has_anchor: bool,
    needs_recalibration: bool,  // 标记是否需要 floor 校准
}
```

#### 三层防御机制

**第一层：高错误率包跳过追踪**

当一个 CCSDS 包中 CRC 错误数 > 50（约 46%），跳过该包的 ptime 追踪，
并调用 `tracker.mark_skip_zone()` 标记需要校准：

```rust
let skip_tracking = n_errors > MAX_CRC_ERRORS_FOR_TRACKING;
if skip_tracking {
    tracker.mark_skip_zone();
}
```

SEC 锚点更新不受影响（`try_update_anchor` 始终调用，自带 utc_tail 校验）。

**第二层：floor 公式校准**

退出高错误区域后，首个事件的 `track_and_compute` 检测到 `needs_recalibration`，
使用 floor 公式独立估算 wrap_count：

```rust
if self.needs_recalibration {
    let raw_delta = ptime as i64 - self.anchor_ptime as i64;
    let raw_delta_seconds = raw_delta as f64 * 2e-6;
    tentative_wrap = ((utc_tail - self.anchor - WRAP_PERIOD - raw_delta_seconds) / WRAP_PERIOD)
        .floor()
        .max(0.0) as i64;
}
```

校准成功后（通过 utc_tail 校验）恢复正常顺序追踪。

**第三层：utc_tail 合理性校验**

所有模式下，计算出的 MET 都需通过 utc_tail 校验：

```
|met - utc_tail - MET_CORRECTION| > 2.0 → 拒绝，不更新状态
```

这是最后的兜底，防止残余幽灵事件或校准误差。

### 6.3 为什么这个方案有效

| 场景 | 处理方式 | 效果 |
|------|----------|------|
| 正常数据 | 顺序追踪 | wrap 边界精确，无 221009A 重叠 |
| 高错误区内 | 跳过追踪 | 幽灵 ptime 不污染 prev_ptime |
| 高错误→正常过渡 | floor 校准 | 正确恢复 wrap_count（±1 wrap 精度） |
| floor 校准 ±1 误差 | 下一个 SEC 锚点重置 | 自动修正 |

## 7. 阈值选择

`MAX_CRC_ERRORS_FOR_TRACKING = 50`（109 个事例中）

- 正常数据包：CRC 错误 < 5，远低于阈值
- 高错误区域：CRC 错误 > 90，远高于阈值
- 50 的阈值提供了充分的安全裕度

## 8. 验证结果

| GRB | 特征 | 结果 |
|-----|------|------|
| 200415A | 无饱和，正常数据 | 1B = 1K 完全一致 (Δ=0) |
| 221009A | 强饱和，FIFO 复位丢数 | 无包重叠，饱和区正确缺失 |
| 260226A | 高 CRC 错误率区域 | 1B ≈ 1K (每 Box Δ≤3)，偏移消除 |

## 附录 B：历史修复尝试记录

## 背景

HXMT HE 1B 数据的时间重建需要处理 19-bit ptime 计数器的回绕（周期 524288 ticks = 1.048576 秒）。

### 关键概念

| 名称 | 含义 |
|------|------|
| ptime | 19-bit 事例计时器 (2μs/tick)，周期 1.048576s |
| anchor | 最近的 Second 事例的 MET (stime + offset) |
| anchor_ptime | 该 Second 事例的 ptime 值 |
| utc_tail | CCSDS 包尾部 4 字节 LE u32，整秒级时间戳 |
| MET_CORRECTION | 1B→1K 时间校正 = 4.0s |
| PTIME_MOD | 524288 (1 << 19) |
| WRAP_PERIOD | 1.048576s |

### 通用公式框架

```
raw_delta = ptime - anchor_ptime  （可为负）
total_ticks = n_wraps × PTIME_MOD + raw_delta
MET = anchor + total_ticks × 2μs + MET_CORRECTION
```

核心问题：**如何确定 n_wraps？**

---

## 问题 1：原始 prev_ptime 方法的缺陷

原始代码使用顺序追踪：
```rust
if ptime < prev_ptime && (prev_ptime - ptime) > HALF_MOD {
    wrap_count += 1;
}
prev_ptime = ptime;
```

**失败场景**：FIFO 溢出后事例时序混乱。例如 Box A pkt 46016：
- evt 0: ptime=1656 (真正的回绕后事例) → wrap_count=1 ✓
- evt 1: ptime=423970 (回绕前事例，FIFO 延迟到达) → 继承 wrap_count=1 ✗

**后果**：~8000 事例/机箱被错误延后 1.048576 秒。

---

## 问题 2：utc_tail 的特性（实测）

通过对比 1K ground truth 和 1B 数据：

| 属性 | 值 |
|------|-----|
| 数据类型 | 整秒 (floor) |
| 更新频率 | ~1 次/秒 |
| 与 anchor 的关系 | 在同一时间尺度 (stime + offset) |
| 与事例真实时间的关系 | 事例 raw_met ∈ [utc_tail, utc_tail + 2) |
| 滞后特性 | anchor 更新后 utc_tail 可能还停留在旧值，最大滞后 ~1-2 秒 |

**关键观测**（Box A 饱和区域）：
```
Pkt 46079-46090: utc_tail=446726292, anchor=446726291 (旧锚点@295)
Pkt 46091-46100: utc_tail=446726292, anchor=446726292 (新锚点@296，utc_tail 未更新！)
Pkt 46101+:      utc_tail=446726293, anchor=446726292 (新锚点，utc_tail 已更新)
```

## 尝试 1：floor((utc_tail - anchor - Δt) / T_wrap) 【当前已提交版本】

```rust
let n_wraps = ((utc_tail - anchor - raw_delta_seconds) / WRAP_PERIOD)
    .floor().max(0.0) as i64;
```

**原理**：utc_tail - anchor 给出从锚点到当前时刻的粗略秒数。减去 raw_delta 后除以回绕周期得到回绕次数。

**结果**：
- ✅ 完美解决了 FIFO 溢出导致的事例乱序问题（pkt 46016 的 evt 1-108）
- ✅ 非饱和区域完全正确
- ❌ 饱和区域残余问题：~978 事例（Box A）从 T+19.0 错位到 T+17.95

**失败原因**：当 anchor 刚更新但 utc_tail 尚未跟进时（pkt 46091-46100），`utc_tail - anchor ≈ 0`，导致 n_wraps 被低估。

**具体失败案例**：
```
anchor = 446726292 (Second@296)
utc_tail = 446726292 (滞后，还没更新到 293)
ptime = 230000, anchor_ptime = 216822
raw_delta = 13178, raw_delta_seconds = 0.026356

n_wraps = floor((292 - 292 - 0.026) / 1.048576) = floor(-0.025) = -1 → max(0) = 0
实际需要 n_wraps = 1 才能到达正确的 T+19 位置
```

## 尝试 2：floor((utc_tail + 1 - anchor - Δt) / T_wrap)

```rust
let n_wraps = ((utc_tail + 1.0 - anchor - raw_delta_seconds) / WRAP_PERIOD)
    .floor().max(0.0) as i64;
```

**原理**：Oracle 建议将 utc_tail 视为 floor(真实时间)，因此事例时间 < utc_tail + 1。

**结果**：
- ❌ 严重破坏非饱和区域
- ❌ 整体差异远大于尝试 1

## 尝试 3：ceil((utc_tail - 1 - anchor - Δt) / T_wrap) 【下界方法】

```rust
let lower = utc_tail - 1.0;
let n_wraps = ((lower - base) / WRAP_PERIOD).ceil().max(0.0) as i64;
```

**原理**：用 utc_tail - 1 作为事例时间的下界，用 ceil 选择最小的合法 n_wraps。

**结果**：
- ✅ 大部分区域正确（T+17.95~19.00 和 T+19.00~19.05 修复）

**失败原因**：下界 utc_tail - 1 对于 anchor 刚更新但 utc_tail 滞后的场景仍然不够。
当 base > lower（即 anchor + raw_delta > utc_tail - 1）时，ceil 返回 0，但实际需要 1。

**原理**：用窗口中心 utc_tail + 0.5 作为目标，四舍五入到最近的回绕次数。

## 最终成功方案 (2026-03-12)

**核心思路**：放弃对 `utc_tail` 绝对值的依赖，改用 SEC 锚点 + 阈值法 + 三遍后处理。

### Pass 1: SEC-anchored 双路径重建

每个事件用最近 SEC 的 `(met, ptime)` 作为锚点。根据 `elapsed = utc_tail - anchor` 选择路径：

#### 正常路径 (elapsed < 1.5×WRAP)
锚点新鲜，至多 1 次 wrap。用阈值法判定：
```rust
if raw_delta < -WRAP_THRESHOLD { raw_delta += PTIME_MOD }
else if raw_delta > PTIME_MOD - WRAP_THRESHOLD { raw_delta -= PTIME_MOD }
```

**优势**：不依赖 utc_tail，免疫 utc_tail 滞后。

#### 过期锚路径 (elapsed ≥ 1.5×WRAP)
FIFO 复位后锚点过期。用 utc_tail 辅助估算 n_wraps，best-of-3 选距 utc_tail 最近的。

**局限**：utc_tail 有偏（反映组装时间），导致部分包高估 +1 wrap。

### Pass 2: fix_wrap_reversals

检测反向跳跃 ≈ -WRAP_PERIOD，用 **gap 判据**区分真假倒退：
- gap > 0.3s → FIFO 复位后真错位 → 整批下移 -WRAP_PERIOD
- gap < 0.3s → 纯文件重排假倒退 → 跳过

### Pass 3: fix_wrap_boundary_dips (新增)

修正 ptime 回绕边界处的错位：

**Step 1 — Dip batch 检测**：
- 扫描正向跳跃 ≈ +WRAP_PERIOD
- 反向查找被 HIGH 包"夹心"的 LOW 批次
- 整批上移 +WRAP_PERIOD，同时修正相邻混合包的 LOW 事件

**Step 2 — 混合包修正**：
- 对 span ≈ WRAP_PERIOD 的包，按邻居层级判定多数/少数簇
- 将少数簇对齐到多数簇 (±WRAP_PERIOD)

### 验证结果

| GRB | Pass 2 触发 | Pass 3 触发 | 逐秒 Δ% |
|-----|-----------|-----------|---------|
| 200415A | 无 | 无 | 0.0% |
| 221009A | 有 (gap判据跳过) | 无 | 0.0% |
| 260226A | 无 | 有 (34包) | 0.0% |

**260226A 修正前后对比**：
- Box A T+19: -5.7% → 0.0%
- Box B T+21/T+24: -4.7%/-8.9% → 0.0%
- Box C T+19/T+24: -5.6%/-8.4% → 0.0%

**关键突破**：
1. 正常路径免疫 utc_tail 滞后
2. Gap 判据区分 FIFO 复位 vs 文件重排
3. 三遍后处理分离不同错位模式，避免相互干扰
