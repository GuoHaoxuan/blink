## 1. 背景

HXMT（慧眼）卫星高能 X 射线望远镜（HE）在极高计数率事件（如伽马射线暴 GRB）期间，探测器前端电子学的数据缓冲区可能溢出，导致部分物理事例（event）数据丢失。我们将这种状态称为**饱和（saturation）**。

准确识别饱和时间区间对科学分析至关重要：
- 饱和区间内的光变曲线不反映真实源流量
- 需要对饱和区间内的计数率进行死时间校正或标记排除
- 1B 级数据在原始管线中不做饱和标注，需要自行判断

## 2. 硬件架构

### 2.1 数据通路概览

```
NaI/CsI 探测器
    ↓ 模拟信号
前端电子学 (ASIC)
    ↓ 数字事例
FPGA
    ↓ 写入
FIFO A (M67204H, 硬件环形缓冲区)
    ↓ MCU读取
MCU (8051 单片机, PDAUA)
    ↓ 打包写入
FIFO B
    ↓ 1553B总线
星务计算机 → 下传
```

### 2.2 FIFO A — M67204H 芯片

- **类型**：硬件 FIFO，环形缓冲区结构
- **关键特性**：
  - 有 Full 标志位（`FIFOAFull1`, `FIFOAFull2`），由硬件自动置位
  - **满时不覆盖**：当 FIFO 满时，FPGA 端写入被阻塞，新事例直接丢弃
  - MCU 可通过写 `ResetFIFOA = 0x00` 复位整个 FIFO
- **地址映射**：`0xE004`（读）/ `0xE001`（复位）

### 2.3 MCU — 8051 单片机

MCU 是**单线程**处理器，通过主循环轮询处理所有任务。

#### 主循环结构（简化）

```c
while(1) {
    ResetWDI = ~ResetWDI;       // 看门狗喂狗
    SearchStack();               // 处理 1553B 消息
    ReStartTimer();              // GPS 同步
    FIFOAFullReset();            // 检查 FIFO A 是否满，满则复位
    
    if (ReadFlag == 1) {         // 每秒一次
        HandleEnginnerLVDS();    // 工程数据处理 (~2ms)
    }
    
    if (INT1Flag == 1) {         // FPGA 中断触发
        HandlePhysicalLVDS();    // 物理数据处理 (~7ms)
        INT1Flag = 0;
        EX1 = 1;                 // 重新使能中断
    }
    
    Refresh();
}
```

#### 关键时间参数

| 操作 | 耗时 | 说明 |
|------|------|------|
| HandlePhysicalLVDS | ~7ms | 从 FIFO A 读 109 个事例，打包写入 FIFO B |
| HandleEnginnerLVDS | ~2ms | 处理工程遥测数据 |
| FIFOAFullReset | 极短 | 检查 Full 标志，触发复位 |

### 2.4 物理数据处理流程 — HandlePhysicalLVDS

```
1. 复位 FIFO B
2. 写入 CCSDS 包头（6 字节）
3. 循环 109 次：
   a. 搜索 0x5A 起始标记
   b. 从 FIFO A 读取 8 字节事例数据
   c. 直接转写入 FIFO B
4. 写入 UTC 时间尾部（4 字节）
5. 通知 FPGA 开始下一轮 LVDS 传输
```

每个 CCSDS 包 = 6 字节包头 + 872 字节载荷（109 × 8 字节事例）+ 4 字节 UTC 尾 = **882 字节**

## 3. 饱和丢数机制

### 3.1 两种丢数模式

#### 模式一：包内无痕丢数（常见）

```
时间线
─────────────────────────────────────────────
FPGA写入:  ████████████████████████████████████
MCU读取:   ████ HandlePhysicalLVDS (~7ms) ████
FIFO状态:  ░░░░░▓▓▓FULL▓▓▓░░░░░░░░░░░░░░░░░░░
                    ↑ 丢弃    ↑ 恢复
```

**过程**：
1. MCU 正在 `HandlePhysicalLVDS()` 中逐字节读取 FIFO A
2. 高计数率下，FPGA 写入速度 > MCU 读出速度
3. FIFO A 在某一刻被写满，FPGA 无法写入，新事例被丢弃
4. MCU 继续读取，读出一些数据后 FIFO A 不再满，FPGA 恢复写入
5. `HandlePhysicalLVDS()` 完成后回到主循环，`FIFOAFullReset()` 检查时 FIFO A 已经不满 → **不触发复位**

**结果**：
- 丢失少量事例
- CCSDS 包结构完整，包间时间间隔无异常
- **无法通过包间 gap 检测到**

**为什么常见**：
- 只要写速率略高于读速率，FIFO 就会在处理包的 7ms 内短暂达到满状态
- 但因为 MCU 一直在读，很快又恢复了

#### 模式二：整包丢失 / FIFO 复位（较少但影响大）

```
时间线
─────────────────────────────────────────────
FPGA写入:  ██████████████████████████████████████
MCU读取:   ████ Handle ████ FIFOAFullReset() ██
FIFO状态:  ░░░░░▓▓▓▓FULL▓▓▓▓▓▓STILL FULL▓▓▓▓▓▓
                                   ↑ 复位！清空所有数据
```

**过程**：
1. 写入速率**极高**，在 `HandlePhysicalLVDS()` 的 7ms 期间写入量远超读出量
2. MCU 处理完一个包回到主循环时，FIFO A **仍然是满的**
3. `FIFOAFullReset()` 检测到 `FIFOAFull1 && FIFOAFull2 == 0`（Full 标志有效）
4. 执行 `ResetFIFOA = 0x00`，**清空整个 FIFO A**
5. 清除 `INT1Flag`，重新使能外部中断

**结果**：
- 丢失整个 FIFO 缓冲区的数据
- 相邻 CCSDS 包之间出现**大的时间间隔**
- **可以通过包间 gap 检测到**

**为什么影响大**：
- 一次复位丢失大量连续事例
- 在光变曲线上表现为计数率突降或时间间隙

### 3.2 对比总结

| | 模式一：包内无痕丢数 | 模式二：FIFO 复位整包丢失 |
|---|---|---|
| **触发条件** | 写速率 > 读速率，但差距不大 | 写速率 >> 读速率 |
| **丢失量** | 少量事例 | 整个 FIFO 缓冲区 |
| **CCSDS 包影响** | 包结构完整 | 包间出现大 gap |
| **可检测性** | ❌ 无法通过包间 gap 检测 | ✅ 可通过包间 gap 检测 |
| **频率** | 饱和期间常见 | 极高计数率时出现 |
| **影响程度** | 较小 | 显著 |

## 4. 饱和区间检测算法

### 4.1 基本思路

通过检测 CCSDS 包之间的**时间间隔异常**来识别发生了 FIFO 复位（模式二）的时间区间。

### 4.2 算法流程

```
┌─────────────────────────────────┐
│  1. 解析所有 CCSDS 包            │
│     parse_events() → 事例列表    │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  2. 时间重建                     │
│     reconstruct_with_wrap_tracking() │
│     → 每个包的 MET 时间列表       │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  3. 计算每个包的时间范围          │
│     PackInfo { min_time, max_time } │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  4. 按 min_time 排序             │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  5. 滑窗检测包间 gap             │
│     gap = next.min_time          │
│         - current.max_time       │
│     if gap > threshold → 饱和区间 │
└─────────────┬───────────────────┘
              ↓
┌─────────────────────────────────┐
│  6. 多机箱合并（A/B/C 取并集）   │
│     任一机箱饱和 → 该时段判定饱和 │
└─────────────────────────────────┘
```

### 4.3 Gap 阈值讨论

当前实现使用固定阈值 `GAP_THRESHOLD = 6.9ms`，但这**不够准确**。

正常包间隔取决于实时事例率 R：

$$T_{packet} \approx \frac{109}{R}$$

| 事例率 R (evt/s) | 正常包间隔 | 6.9ms 阈值表现 |
|------------------|-----------|---------------|
| 10,000 | ~10.9ms | ⚠️ 误判：正常间隔已超过阈值 |
| 15,000 | ~7.3ms | ⚠️ 边界情况 |
| 20,000 | ~5.5ms | ✅ 能区分 |
| 50,000 | ~2.2ms | ✅ 灵敏度足够 |

**改进方向**：
- 使用自适应阈值，基于局部事例率动态计算
- 或直接对比相邻包间隔与该时段平均包间隔的偏离程度

### 4.4 时间重建算法

饱和区间检测的前提是准确的事例时间重建。

#### 事例类型

每个 8 字节事例通过 CRC 校验和类型标志位（bit 5:4）区分：

| 类型 | 标志位 | 内容 |
|------|--------|------|
| Event | 0x00 / 0x20 | 物理事例：channel + ptime |
| Second | 0x10 | 秒标记：stime（整秒 MET）+ ptime |
| Error | 其他 | CRC 校验失败 |

#### 重建公式

$$MET = met\_anchor + (wrap\_count \times 524288 + ptime - anchor\_ptime) \times 2\mu s + 4.0s$$

其中：
- `met_anchor = stime + offset`（来自最近的有效 Second 事例）
- `ptime`：19 位精确时间计数器（2μs 分辨率）
- `wrap_count`：ptime 回绕次数（回绕周期 = 2^19 × 2μs ≈ 1.048576s）
- `4.0s`：1B → 1K 经验校正

#### 已知问题

饱和间隙后恢复时，Second 事例的 `ptime` 固定为 0，导致假回绕检测：

```
间隙前: Event ptime=500000
间隙后: Second ptime=0  ← 被误判为回绕
        Event ptime=100  ← 正常递增
```

结果：多计一次回绕 → MET 偏移 +1.048576s

**修复方向**：分离 `anchor_ptime`（用于 MET 计算）和 `chain_ptime`（用于 Event 间的回绕检测），避免 Second 的 `ptime=0` 污染回绕判断。

## 5. 三机箱合并策略

HE 望远镜有三个独立机箱（Box A / B / C），每个机箱有独立的 FIFO A 和 MCU。

判定策略：**任一机箱饱和 → 该时间段判定为饱和**

```rust
// 合并算法
1. 分别对三个机箱扫描饱和区间
2. 收集所有区间，按起始时间排序
3. 合并重叠区间（取并集）
4. 最终得到统一的饱和时间段列表
```

判断某一时刻是否饱和时，使用**二分查找**在已排序的区间列表中定位。

## 8. 静默丢数检测（包内无痕丢数）

### 8.1 原理

模式一（包内无痕丢数）无法通过包间 gap 检测，但可以通过**包内事件间隔统计异常**来发现。

在 FIFO 拥塞期间，MCU 读速率（~15797 evt/s）成为瓶颈，事件以近乎均匀的间隔被读出。如果 FIFO 在读取过程中短暂满溢后恢复，丢弃的事件会在包内产生一个异常大的时间间隔。

### 8.2 检测方法

对每个高事件率包（包内事件率 > MCU 读速率 15797 evt/s）：

1. 计算包内事件间隔序列 `dt[i] = t[i+1] - t[i]`
2. 用过滤后的间隔（< 1ms）估算包内真实事件率 `λ = 1 / mean(dt_filtered)`
3. 对每个间隔计算泊松概率 `log₁₀(p) = -λ × dt / ln(10)`
4. 若 `log₁₀(p) < -10`，标记为静默丢数可疑位置

**关键改进**：lambda 必须用过滤后的间隔估算，否则大 gap 本身会拉低 lambda，导致同包内其他较小的异常间隔漏检。

### 8.3 检测结果

| GRB | Box A | Box B | Box C | 合计 suspects | 独立包数 |
|-----|-------|-------|-------|--------------|---------|
| 200415A | 1 | 0 | 0 | **1** | 1 |
| 221009A | 42 | 42 | 39 | **138** | ~90 |
| 260226A | 0 | 0 | 0 | **0** | 0 |

### 8.4 典型特征

- **位置集中在包头/包尾**：大多数异常间隔出现在 `evt_gap_idx=0`（包头）或 `evt_gap_idx=106`（包尾），对应 MCU 在包边界处开始/结束读取 FIFO 的时刻，此时 FIFO 写入积压最容易发生满溢
- **间隔规模**：典型 500μs ~ 6000μs（正常间隔 ~10-60μs），对应丢失数十到数百个事件
- **与 FIFO 复位的关系**：静默丢数集中在 FIFO 复位（模式二）密集区附近，说明两种模式是同一拥塞过程的不同严重程度表现
- **GRB 强度相关**：221009A（史上最亮 GRB）有大量静默丢数，260226A（较弱）无静默丢数

### 8.5 可视化工具

- `analyze_silent_drop.py`：扫描所有包，检测异常间隔，输出 suspects JSON
- `plot_silent_drop.py`：为每个可疑包画三面板图（概览 ±100ms、放大到包级、包内间隔柱状图），同一包内多个异常间隔合并显示
